<h2>8. Обґрунтування вибору типу бази даних</h2>

<h3>8.1. Вступ</h3>

<p>
Бази даних є ключовим компонентом будь-якої інформаційної системи, особливо для онлайн-банкінгу, де обробка фінансових операцій вимагає високої надійності, узгодженості та швидкодії. Вибір правильної СУБД впливає на ефективність роботи системи, її масштабованість та здатність інтегруватися з зовнішніми сервісами.
</p>

<p>Основні аспекти, що визначають вибір СУБД для онлайн-банкінгу:</p>
<ul>
    <li>Характер даних: структуровані таблиці або напівструктуровані документи.</li>
    <li>Транзакційні вимоги: забезпечення атомарності, узгодженості, ізольованості та довговічності (ACID).</li>
    <li>Продуктивність: швидкість обробки транзакцій і запитів при великій кількості користувачів.</li>
    <li>Масштабованість та гнучкість: здатність системи зростати горизонтально та вертикально.</li>
    <li>Надійність і резервування: можливість відновлення даних після збоїв або втрати серверів.</li>
    <li>Безпека: контроль доступу, шифрування чутливих даних, аудит операцій.</li>
</ul>

<hr>

<h3>8.2. Порівняння моделей збереження даних</h3>

<h4>8.2.1. Реляційні бази даних (SQL)</h4>

<ul>
    <li>Структуровані таблиці: кожна сутність зберігається у вигляді таблиці з визначеними стовпцями та типами даних.</li>
    <li>ACID транзакції: гарантують правильність фінансових операцій, збереження послідовності та уникнення втрат даних.</li>
    <li>Сильна узгодженість: критично для банківської системи, де будь-яка помилка у балансі рахунку неприпустима.</li>
    <li>Можливості складних запитів: підтримка JOIN, агрегатних функцій, індексування та тригерів.</li>
    <li>Підходить для структурованих даних і систем, де критична точність обчислень та взаємозв’язків між об’єктами.</li>
</ul>

<h4>8.2.2. NoSQL (документні, key-value, графові бази)</h4>

<ul>
    <li>Гнучка структура даних: зручна для великих обсягів неструктурованих або напівструктурованих даних.</li>
    <li>Висока продуктивність і масштабованість: горизонтальне масштабування і розподілене зберігання.</li>
    <li>Обмежена узгодженість: часто використовується eventual consistency, що не підходить для фінансових транзакцій.</li>
    <li>Швидкий доступ до даних у кешованих або аналітичних сценаріях: Redis для кешування балансів, MongoDB для великих логів операцій.</li>
    <li>Найкраще підходить для аналітичних задач та обробки великих потоків подій.</li>
</ul>

<hr>

<h3>8.3. Вибір конкретної СУБД</h3>

<p>
Для системи онлайн-банкінгу оптимальною є реляційна база даних <strong>PostgreSQL</strong>:
</p>

<ul>
    <li>ACID транзакції: гарантують надійне проведення переказів і збереження даних користувачів.</li>
    <li>Складні зв’язки між сутностями: користувачі, рахунки, транзакції, картки, журнали аудиту.</li>
    <li>Масштабованість: підтримка реплікації та шардінгу для обробки великих навантажень.</li>
    <li>Вбудовані механізми безпеки: шифрування даних на диску, контроль доступу до таблиць та рядків, аудит змін.</li>
    <li>Підтримка структурованих та напівструктурованих даних: JSON для інтеграції з мікросервісами.</li>
    <li>Надійність: регулярні резервні копії, підтримка high availability (кластеризація Master-Replica).</li>
</ul>

<p>
PostgreSQL поєднує високу надійність, безпеку, узгодженість і масштабованість, що критично для фінансових систем.
</p>
