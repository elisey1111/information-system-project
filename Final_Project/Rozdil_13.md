# <h4>13.2.2. Мета проєкту</h4>

# 

# <p>

# Метою проєкту є розробка архітектурної моделі інформаційної системи онлайн-банкінгу з обґрунтуванням вибору технологій, бази даних, методології розробки, механізмів безпеки та масштабування, а також побудова формалізованих моделей (UML, ER) для підтвердження структурної та функціональної цілісності рішення.

# </p>

# 

# <h4>13.2.3. Завдання проєкту</h4>

# &nbsp;   <li>Визначити функціональні та нефункціональні вимоги.</li>

# &nbsp;   <li>Розробити архітектуру системи з поділом на рівні.</li>

# &nbsp;   <li>Побудувати модель даних.</li>

# &nbsp;   <li>Обґрунтувати вибір СУБД.</li>

# &nbsp;   <li>Проаналізувати масштабованість.</li>

# &nbsp;   <li>Розробити модель безпеки.</li>

# &nbsp;   <li>Визначити перспективи розвитку.</li>

# &nbsp;   <li>Обрати методологію розробки.</li>

# &nbsp;   <li>Оформити результати у вигляді завершеної проєктної документації.</li>

# </ul>

# <h3>13.3. Основні розділи документації</h3>

# 

# <p>Проєктна документація структурована відповідно до пунктів 1–12 та містить такі змістові блоки:</p>

# 

# <ol>

# &nbsp;   <li>

# &nbsp;       <strong>Аналіз предметної області</strong> — описано специфіку онлайн-банкінгу, основні бізнес-процеси, типових користувачів та взаємодію між підсистемами.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Визначення вимог</strong> — сформульовано функціональні вимоги (реєстрація, авторизація, транзакції, звітність) та нефункціональні (безпека, продуктивність, масштабованість, доступність).

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Архітектура системи</strong> — розроблено багаторівневу архітектуру (Presentation Layer, Business Logic Layer, Data Layer) з визначенням залежностей і інтеграційних точок.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Модель даних</strong> — побудовано ER-модель із визначенням сутностей, атрибутів, первинних і зовнішніх ключів, зв’язків між таблицями.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>UML-моделювання</strong> — розроблено діаграми: Use Case, класів, компонентів, розгортання.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Обґрунтування вибору СУБД</strong> — аргументовано вибір PostgreSQL з огляду на ACID-властивості, транзакційність та складні зв’язки.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Аналіз масштабованості</strong> — розглянуто вертикальне та горизонтальне масштабування, балансування навантаження, кешування та кластеризацію.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Система безпеки</strong> — описано механізми автентифікації (2FA), модель RBAC, шифрування даних, резервне копіювання та контроль цілісності.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Перспективи розвитку</strong> — визначено напрями еволюції: мікросервіси, Open Banking, мобільні застосунки, AI для аналізу ризиків.

# &nbsp;   </li>

# &nbsp;   <li>

# &nbsp;       <strong>Методологія розробки</strong> — обґрунтовано вибір Agile/Scrum як найбільш відповідного підходу для еволюційної архітектури системи.

# &nbsp;   </li>

# </ol>

#

# <h3>13.4. Додатки</h3>

# 

# <p>До документації включено додатки, що підтверджують формалізованість проєкту:</p>

# <ul>

# &nbsp;   <li><strong>Додаток А — UML-діаграми:</strong> Use Case, Class, Component, Deployment Diagram</li>

# &nbsp;   <li><strong>Додаток B — ER-діаграма бази даних:</strong> структура таблиць, ключі та зв’язки</li>

# &nbsp;   <li><strong>Додаток C — Архітектурна схема:</strong> графічна модель взаємодії рівнів системи</li>

# &nbsp;   <li><strong>Додаток D — Схема інтеграцій:</strong> взаємодія з зовнішніми платіжними та аналітичними сервісами</li>

# </ul>

# 

# <p>Усі діаграми виконані відповідно до стандартів UML 2.x та принципів структурного моделювання.</p>
