# <h2>12. Обрати методологію розробки</h2>

# 

# <h3>12.1. Вступ</h3>

# 

# <p>

# Вибір методології розробки програмного забезпечення є стратегічним рішенням, яке безпосередньо впливає на якість кінцевого продукту, терміни реалізації, керованість ризиками та ефективність командної взаємодії.

# </p>

# 

# <p>Для системи онлайн-банкінгу характерні:</p>

# <ul>

# &nbsp;   <li>високі вимоги до безпеки та надійності;</li>

# &nbsp;   <li>складна бізнес-логіка;</li>

# &nbsp;   <li>необхідність інтеграції з зовнішніми фінансовими сервісами;</li>

# &nbsp;   <li>можливі зміни нормативних вимог;</li>

# &nbsp;   <li>потреба у поступовому розширенні функціоналу.</li>

# </ul>

# 

# <p>

# З урахуванням цих факторів обрано Agile-підхід із використанням фреймворку Scrum як основну методологію розробки.

# </p>

# 

# <hr>

# 

# <h3>12.2. Аналіз можливих методологій</h3>

# 

# <p>

# Перед обґрунтуванням вибору було розглянуто основні підходи до розробки програмного забезпечення.

# </p>

# 

# <h4>12.2.1. Каскадна модель (Waterfall)</h4>

# 

# <p>Каскадний підхід передбачає послідовне проходження етапів:</p>

# 

# <ul>

# &nbsp;   <li>Аналіз вимог</li>

# &nbsp;   <li>Проєктування</li>

# &nbsp;   <li>Реалізація</li>

# &nbsp;   <li>Тестування</li>

# &nbsp;   <li>Впровадження</li>

# &nbsp;   <li>Підтримка</li>

# </ul>

# 

# <p><strong>Переваги:</strong></p>

# <ul>

# &nbsp;   <li>чітка документація;</li>

# &nbsp;   <li>зрозуміла структура;</li>

# &nbsp;   <li>передбачуваність етапів.</li>

# </ul>

# 

# <p><strong>Недоліки для даного проєкту:</strong></p>

# <ul>

# &nbsp;   <li>складність внесення змін після завершення етапу;</li>

# &nbsp;   <li>низька гнучкість при зміні вимог;</li>

# &nbsp;   <li>високі ризики при довготривалих проєктах.</li>

# </ul>

# 

# <p>

# Для системи онлайн-банкінгу, де вимоги можуть змінюватися (законодавство, інтеграції, функціональні доповнення), каскадна модель є недостатньо гнучкою.

# </p>

# 

# <h4>12.2.2. Спіральна модель</h4>

# 

# <p>

# Спіральна модель орієнтована на управління ризиками через багаторазові цикли планування, аналізу та прототипування.

# </p>

# 

# <p><strong>Переваги:</strong></p>

# <ul>

# &nbsp;   <li>сильний акцент на аналізі ризиків;</li>

# &nbsp;   <li>підходить для складних і критичних систем.</li>

# </ul>

# 

# <p><strong>Недоліки:</strong></p>

# <ul>

# &nbsp;   <li>висока складність управління;</li>

# &nbsp;   <li>значні витрати на планування;</li>

# &nbsp;   <li>надмірність для середніх команд.</li>

# </ul>

# 

# <p>

# Для академічного або середнього комерційного проєкту така модель може бути надто складною в реалізації.

# </p>

# 

# <h4>12.2.3. RUP (Rational Unified Process)</h4>

# 

# <p>

# RUP є формалізованим ітеративним підходом із чіткими фазами:

# </p>

# 

# <ul>

# &nbsp;   <li>Inception</li>

# &nbsp;   <li>Elaboration</li>

# &nbsp;   <li>Construction</li>

# &nbsp;   <li>Transition</li>

# </ul>

# 

# <p><strong>Переваги:</strong></p>

# <ul>

# &nbsp;   <li>формалізованість;</li>

# &nbsp;   <li>сильна орієнтація на архітектуру;</li>

# &nbsp;   <li>детальна документація.</li>

# </ul>

# 

# <p><strong>Недоліки:</strong></p>

# <ul>

# &nbsp;   <li>складність впровадження;</li>

# &nbsp;   <li>потребує значних ресурсів;</li>

# &nbsp;   <li>висока формалізація процесів.</li>

# </ul>

# 

# <p>

# Для невеликої або середньої команди це може бути надмірним.

# </p>

# 

# <hr>

# 

# <h3>12.3. Обґрунтування вибору Agile / Scrum</h3>

# 

# <h4>12.3.1. Причини вибору Agile</h4>

# 

# <ul>

# &nbsp;   <li>адаптивність до змін;</li>

# &nbsp;   <li>ітеративність розробки;</li>

# &nbsp;   <li>швидке отримання робочого інкременту продукту;</li>

# &nbsp;   <li>активна взаємодія із замовником;</li>

# &nbsp;   <li>пріоритет робочого програмного забезпечення над надмірною документацією.</li>

# </ul>

# 

# <p>

# Для системи онлайн-банкінгу це є критично важливим, оскільки вимоги можуть уточнюватися або змінюватися, необхідно швидко реагувати на зміни регуляторних вимог, а функціонал розширюється поступово.

# </p>

# 

# <h4>12.3.2. Вибір Scrum як реалізації Agile</h4>

# 

# <p>

# Scrum є структурованим фреймворком у межах Agile, який передбачає:

# </p>

# 

# <ul>

# &nbsp;   <li>поділ роботи на короткі ітерації (спринти) тривалістю 2–4 тижні;</li>

# &nbsp;   <li>чітко визначені ролі;</li>

# &nbsp;   <li>регулярні зустрічі;</li>

# &nbsp;   <li>постійну демонстрацію результату.</li>

# </ul>

# 

# <hr>

# 

# <h3>12.4. Узгодженість Scrum зі структурою системи</h3>

# 

# <p>

# Архітектура системи побудована за принципом:

# </p>

# 

# <ul>

# &nbsp;   <li>Presentation Layer</li>

# &nbsp;   <li>Business Logic Layer</li>

# &nbsp;   <li>Data Layer</li>

# </ul>

# 

# <p>

# Також передбачено можливий перехід до мікросервісної архітектури.

# </p>

# 

# <h4>12.4.1. Модульність системи</h4>

# 

# <ul>

# &nbsp;   <li>модуль автентифікації;</li>

# &nbsp;   <li>модуль управління рахунками;</li>

# &nbsp;   <li>модуль транзакцій;</li>

# &nbsp;   <li>модуль звітності;</li>

# &nbsp;   <li>модуль адміністративного управління.</li>

# </ul>

# 

# <p>

# Кожен модуль може розроблятися як окремий інкремент у межах спринту, що повністю відповідає ітеративному характеру Scrum.

# </p>

# 

# <h4>12.4.2. Інкрементальний розвиток функціоналу</h4>

# 

# <p><strong>MVP:</strong></p>

# <ul>

# &nbsp;   <li>реєстрація;</li>

# &nbsp;   <li>авторизація;</li>

# &nbsp;   <li>перегляд рахунку.</li>

# </ul>

# 

# <p><strong>Подальше розширення:</strong></p>

# <ul>

# &nbsp;   <li>перекази;</li>

# &nbsp;   <li>інтеграція з платіжними сервісами;</li>

# &nbsp;   <li>аналітика;</li>

# &nbsp;   <li>масштабування та оптимізація.</li>

# </ul>

# 

# <hr>

# 

# <h3>12.5. Узгодженість із командною організацією роботи</h3>

# 

# <h4>12.5.1. Ролі у Scrum</h4>

# 

# <ul>

# &nbsp;   <li><strong>Product Owner</strong> – формує backlog, визначає пріоритети, відповідає за бізнес-цінність.</li>

# &nbsp;   <li><strong>Scrum Master</strong> – контролює дотримання процесу, усуває перешкоди, координує команду.</li>

# &nbsp;   <li><strong>Development Team</strong> – backend-розробники, frontend-розробники, тестувальники, DevOps.</li>

# </ul>

# 

# <h4>12.5.2. Організація процесу</h4>

# 

# <ul>

# &nbsp;   <li>Sprint Planning — планування завдань;</li>

# &nbsp;   <li>Daily Scrum — щоденні короткі зустрічі;</li>

# &nbsp;   <li>Sprint Review — демонстрація результату;</li>

# &nbsp;   <li>Sprint Retrospective — аналіз процесу та покращення.</li>

# </ul>

# 

# <p>

# Це забезпечує прозорість процесу, постійний контроль якості, швидке виявлення проблем та адаптацію до змін.

# </p>

# 

# <hr>

# 

# <h3>12.6. Управління ризиками в межах Scrum</h3>

# 

# <p>

# Хоча Scrum не є формально ризик-орієнтованою моделлю, ризики знижуються за рахунок:

# </p>

# 

# <ul>

# &nbsp;   <li>коротких циклів розробки;</li>

# &nbsp;   <li>раннього тестування;</li>

# &nbsp;   <li>постійної інтеграції;</li>

# &nbsp;   <li>регулярного зворотного зв’язку.</li>

# </ul>

# 

# <p>

# Це дозволяє оперативно виявляти проблеми безпеки, помилки бізнес-логіки та недоліки інтеграцій.

# </p>

# 

# <hr>

# 

# <h3>12.7. Переваги обраної методології для онлайн-банкінгу</h3>

# 

# <ul>

# &nbsp;   <li>Гнучкість при зміні вимог.</li>

# &nbsp;   <li>Можливість поетапного масштабування.</li>

# &nbsp;   <li>Швидкий випуск функціональних оновлень.</li>

# &nbsp;   <li>Зменшення технічного боргу.</li>

# &nbsp;   <li>Підвищення прозорості процесу.</li>

# &nbsp;   <li>Краща адаптація до еволюційної архітектури системи.</li>

# </ul>

# 

# <hr>

# 

# <h3>12.8. Висновок</h3>

# 

# <p>

# Для розробки системи онлайн-банкінгу обрано Agile-підхід із використанням Scrum, оскільки він відповідає модульній архітектурі системи, підтримує еволюційний розвиток, дозволяє швидко адаптуватися до змін, забезпечує ефективну командну взаємодію та мінімізує ризики при розробці складної фінансової системи.

# </p>

# 

# <p>

# Обрана методологія створює умови для стабільного, контрольованого та масштабованого розвитку системи в довгостроковій перспективі.

# </p>
