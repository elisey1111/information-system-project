# <h2>10. Проаналізувати систему безпеки</h2>

# 

# <h3>10.1. Вступ</h3>

# 

# <p>

# Інформаційна безпека є критичною складовою будь-якої сучасної системи онлайн-банкінгу. Система безпеки має забезпечувати захист від несанкціонованого доступу, збереження конфіденційності персональних та фінансових даних клієнтів, гарантувати цілісність транзакцій та надійність функціонування навіть у випадку технічних збоїв або атак.

# </p>

# 

# <p>

# Для досягнення цих цілей у системі впроваджуються комплексні механізми автентифікації, авторизації, управління правами доступу, шифрування даних, політика резервного копіювання та контроль цілісності.

# </p>

# 

# <hr>

# 

# <h3>10.2. Автентифікація користувачів</h3>

# 

# <p>

# Автентифікація забезпечує підтвердження особи користувача та є першим рівнем захисту системи. Для онлайн-банкінгу застосовується багаторівневий підхід:

# </p>

# 

# <h4>10.2.1. Класична автентифікація за логіном і паролем</h4>

# 

# <ul>

# &nbsp;   <li>Паролі зберігаються у хешованому вигляді з використанням bcrypt або Argon2 з унікальною сіллю (salt).</li>

# &nbsp;   <li>Сіль захищає від атак типу rainbow table та робить кожен хеш унікальним навіть при однакових паролях у різних користувачів.</li>

# &nbsp;   <li>Політика складності паролю: мінімальна довжина, обов’язкове включення великих літер, цифр та спецсимволів, регулярна зміна паролю.</li>

# </ul>

# 

# <h4>10.2.2. Двофакторна автентифікація (2FA)</h4>

# 

# <ul>

# &nbsp;   <li>Другий рівень перевірки: одноразові коди (OTP), SMS або push-нотифікації через додатки.</li>

# &nbsp;   <li>Зменшує ризик компрометації облікового запису навіть при викраденні пароля.</li>

# </ul>

# 

# <h4>10.2.3. Біометрична автентифікація</h4>

# 

# <ul>

# &nbsp;   <li>Інтеграція сканерів відбитків пальців або технологій розпізнавання обличчя у мобільних клієнтах.</li>

# &nbsp;   <li>Використовується як додатковий фактор для підвищення безпеки великих операцій.</li>

# </ul>

# 

# <hr>

# 

# <h3>10.3. Авторизація та управління правами доступу</h3>

# 

# <h4>10.3.1. Модель доступу</h4>

# 

# <p>Використовується Role-Based Access Control (RBAC), де доступ надається залежно від ролі користувача:</p>

# <ul>

# &nbsp;   <li><strong>Клієнт:</strong> доступ до рахунків, історії транзакцій, платежів і переказів, управління картками.</li>

# &nbsp;   <li><strong>Менеджер банку:</strong> підтвердження великих транзакцій, доступ до звітів клієнтів, обробка кредитних запитів.</li>

# &nbsp;   <li><strong>Адміністратор системи:</strong> управління обліковими записами, налаштування системи, резервне копіювання, моніторинг безпеки.</li>

# </ul>

# 

# <h4>10.3.2. Механізми обмеження доступу</h4>

# 

# <ul>

# &nbsp;   <li>Контроль на рівні функцій (Feature-Level Access Control).</li>

# &nbsp;   <li>Контроль на рівні даних (Row-Level Security) — доступ до конкретних записів лише для відповідних користувачів.</li>

# &nbsp;   <li>Журналювання дій (Audit Logging) — усі дії користувачів і адміністраторів фіксуються для відстеження порушень безпеки.</li>

# </ul>

# 

# <hr>

# 

# <h3>10.4. Захист і шифрування чутливих даних</h3>

# 

# <h4>10.4.1. Зберігання даних</h4>

# 

# <ul>

# &nbsp;   <li>Конфіденційні дані (паспортні дані, реквізити карток) зберігаються у зашифрованому вигляді (AES-256).</li>

# &nbsp;   <li>Хешування паролів та облікових даних за сучасними стандартами.</li>

# </ul>

# 

# <h4>10.4.2. Передача даних</h4>

# 

# <ul>

# &nbsp;   <li>TLS 1.2/1.3 для захисту при передачі через мережу (HTTPS).</li>

# &nbsp;   <li>Шифрування сесійних токенів та cookie для запобігання викраденню сесії.</li>

# </ul>

# 

# <h4>10.4.3. Обмеження доступу</h4>

# 

# <ul>

# &nbsp;   <li>Розділення доступу на рівні бази даних та інтерфейсів (користувачі, оператори, адміністратори).</li>

# &nbsp;   <li>Принцип least privilege — мінімальний набір прав для виконання завдань.</li>

# </ul>

# 

# <hr>

# 

# <h3>10.5. Політика резервного копіювання</h3>

# 

# <ul>

# &nbsp;   <li>Повні резервні копії: щодня вночі.</li>

# &nbsp;   <li>Інкрементальні копії: кожні 1–2 години для мінімальних втрат даних.</li>

# &nbsp;   <li>Відновлення по точці часу (Point-in-time recovery).</li>

# &nbsp;   <li>Зберігання копій у віддаленому дата-центрі для відмовостійкості.</li>

# &nbsp;   <li>Шифрування резервних копій для захисту даних.</li>

# </ul>

# 

# <hr>

# 

# <h3>10.6. Контроль цілісності даних</h3>

# 

# <ul>

# &nbsp;   <li>Контрольні суми (Checksum, Hash) для критичних файлів та транзакцій.</li>

# &nbsp;   <li>Транзакційна цілісність (ACID) у PostgreSQL — операції виконуються повністю або відкотяться.</li>

# &nbsp;   <li>Регулярні аудити бази даних і журналів для виявлення несанкціонованих змін.</li>

# </ul>

# 

# <h3>10.7. Інструменти моніторингу та реагування</h3>

# 

# <ul>

# &nbsp;   <li>Системи моніторингу безпеки (SIEM) для відстеження аномальних подій.</li>

# &nbsp;   <li>Повідомлення про інциденти: автоматичне сповіщення адміністраторів.</li>

# &nbsp;   <li>План реагування на інциденти: локалізація, усунення та відновлення після атак.</li>

# </ul>

# 

# <h3>10.8. Висновок</h3>

# 

# <p>

# Розроблена система безпеки є багаторівневою та комплексною: багатофакторна автентифікація, рольова модель доступу, шифрування даних у базі та під час передачі, політика резервного копіювання та контроль цілісності, моніторинг і аудит. Це гарантує надійний захист персональних і фінансових даних, стійкість системи до загроз та відповідність сучасним стандартам безпеки для онлайн-банкінгу.

# </p>
