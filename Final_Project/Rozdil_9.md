# <h2>9. Аналіз можливостей масштабування системи</h2>

# 

# <h3>9.1. Вступ</h3>

# 

# <p>

# Масштабування системи онлайн-банкінгу забезпечує її стійкість, високу продуктивність і безперервну доступність навіть при різкому збільшенні кількості користувачів або транзакцій.

# </p>

# 

# <p>Масштабування включає:</p>

# <ul>

# &nbsp;   <li>Вертикальне масштабування – збільшення ресурсів одного сервера.</li>

# &nbsp;   <li>Горизонтальне масштабування – додавання нових серверів для розподілу навантаження.</li>

# &nbsp;   <li>Кешування і балансування навантаження – оптимізація швидкості обробки запитів.</li>

# &nbsp;   <li>Кластеризація бази даних і реплікація – забезпечення відмовостійкості та доступності даних.</li>

# </ul>

# 

# <hr>

# 

# <h3>9.2. Вертикальне масштабування</h3>

# 

# <p>

# Принцип: додати CPU, оперативну пам’ять або швидкі SSD на одному сервері.

# </p>

# 

# <ul>

# &nbsp;   <li>Переваги: простота, мінімальна зміна архітектури.</li>

# &nbsp;   <li>Недоліки: обмежена фізичними характеристиками сервера; при виході сервера з ладу система стає недоступною.</li>

# &nbsp;   <li>Використання: підходить для початкових етапів проєкту або при середньому навантаженні.</li>

# </ul>

# 

# <h3>9.3. Горизонтальне масштабування</h3>

# 

# <ul>

# &nbsp;   <li>Принцип: додавання нових серверів або вузлів для обробки додаткового навантаження.</li>

# &nbsp;   <li>Переваги: висока доступність, можливість обробки тисяч одночасних запитів.</li>

# &nbsp;   <li>Складність: потребує балансування запитів, синхронізації даних, управління станом сесій.</li>

# &nbsp;   <li>Використання: необхідно для великих систем з високою кількістю транзакцій та користувачів.</li>

# </ul>

# 

# <h3>9.4. Балансування навантаження</h3>

# 

# <ul>

# &nbsp;   <li>Load Balancer (Nginx, HAProxy): рівномірно розподіляє трафік між веб-серверами.</li>

# &nbsp;   <li>База даних: Master-Replica конфігурація — Master для запису, репліки для читання.</li>

# &nbsp;   <li>Переваги: запобігає перевантаженню одного вузла, забезпечує безперервність роботи.</li>

# </ul>

# 

# <h3>9.5. Резервування даних та кластеризація</h3>

# 

# <ul>

# &nbsp;   <li>Реплікація PostgreSQL: автоматичне копіювання даних на додаткові вузли.</li>

# &nbsp;   <li>High Availability: автоматичне перемикання на репліку при відмові Master.</li>

# &nbsp;   <li>Регулярне резервне копіювання: дозволяє відновити систему у випадку апаратного збою або втрати даних.</li>

# </ul>

# 

# <h3>9.6. Використання мікросервісної архітектури</h3>

# 

# <ul>

# &nbsp;   <li>Розподіл системи на незалежні сервіси: User Service, Transaction Service, Card Service, Notification Service.</li>

# &nbsp;   <li>Кожен сервіс можна горизонтально масштабувати окремо.</li>

# &nbsp;   <li>Зменшує навантаження на центральний сервер і підвищує гнучкість системи.</li>

# </ul>

# 

# <h3>9.7. Кешування</h3>

# 

# <ul>

# &nbsp;   <li>Redis або Memcached: кешування результатів запитів (баланс рахунку, останні транзакції).</li>

# &nbsp;   <li>Зменшує навантаження на базу даних і прискорює обробку запитів користувачів.</li>

# </ul>

# 

# <h3>9.8. Висновок</h3>

# 

# <p>

# Система спроектована з урахуванням вертикального та горизонтального масштабування, балансування запитів, кластеризації бази даних і кешування.

# </p>

# 

# <p>

# Вибір PostgreSQL як основної СУБД забезпечує надійність, узгодженість та безпеку фінансових даних.

# </p>

# 

# <p>

# Архітектура дозволяє ефективно обробляти великий потік користувачів, одночасні транзакції та масштабуватися у майбутньому без втрати продуктивності.

# </p>
